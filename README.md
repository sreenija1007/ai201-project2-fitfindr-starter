# FitFindr Agent

FitFindr is a helpful AI styling assistant. It interprets what a user wants to buy, scans a database for matching thrifted clothes, checks the user's current closet to see how they can wear it, and writes a fun social media post about the final look.


## Tool Inventory

### 1. `search_listings`
* **Purpose:** Scans the JSON data to find clothing items that fit the user's request and budget.
* **Inputs:** * `description (str)`: Text describing the type of clothing or aesthetic.
  * `size (str)`: The requested clothing size.
  * `max_price (float)`: The top dollar amount the user wants to spend.
* **Outputs:** `list[dict]` - An array of item objects that match the criteria.

### 2. `suggest_outfit`
* **Purpose:** Takes the top item found from the database and uses an LLM to build a cool outfit out of it and the user's existing wardrobe.
* **Inputs:** * `new_item (dict)`: The specific item object chosen from the search step.
  * `wardrobe (dict)`: The object containing all items currently in the user's closet.
* **Outputs:** `str` - A text string containing personalized styling recommendations.

### 3. `create_fit_card`
* **Purpose:** Takes the outfit ideas and transforms them into a short, casual caption that is ready to share on social media.
* **Inputs:**
  * `outfit (str)`: The outfit text generated by the previous styling tool.
  * `new_item (dict)`: The item object used to inject things like the price or item name.
* **Outputs:** `str` - A brief text string written in a casual style with emojis.

## The Planning Loop
The agent bases its next steps entirely on what the search tool returns. First, it triggers `search_listings`. It evaluates the length of the returned array of items. **If the length is zero**, it completely cuts off the rest of the workflow, writes a helpful message to the session error path, and finishes early. **If it finds matches**, it grabs the first item, saves it to memory, and runs the outfit suggestion tool followed immediately by the fit card tool.

## State Management Approach
We use a single session state dictionary that lives through the whole interaction. When `search_listings` finds an item, the agent drops it into `session['selected_item']`. The outfit tool reads its inputs right out of `session['selected_item']` and `session['wardrobe']`. The text output from the outfit tool then gets saved into `session['outfit_suggestion']` so the fit card tool can easily pull it. This means data flows cleanly from one tool to the next without the user ever having to repeat themselves.

## Error Handling
* **Search Tool:** If there are no results (e.g., when testing the "designer ballgown size XXS under $5" query), the tool returns an empty array. The agent notices this, halts the process, and returns a friendly message telling the user to try dropping the price filter.
* **Outfit Tool:** If the user has an empty closet, the tool ignores the empty closet file and suggests styling the new piece with simple items like plain denim or white tees so the app doesn't crash.
* **Fit Card Tool:** If the styling text input is missing, the tool generates a quick caption about the thrift find itself while leaving out the missing styling text and joking that the styling AI is taking a break.

## Spec Reflection
The `planning.md` file was highly effective for mapping out the architecture. Drawing the ASCII flow diagram made it very obvious where the workflow needed to break off if the search failed. During implementation, I diverged slightly from my original plan by using manual string-splitting and array iterations to extract the user's size and budget limits from their prompt, rather than relying on an external regex library, to keep the code straightforward and readable.

## AI Usage Details
1. **Tool Implementation:** I fed ChatGPT the specific parameters, return expectations, and failure scenarios for my three tools from my `planning.md`. I told it to write the code for `tools.py` using the data loader helper functions. It successfully connected to the Groq API, but I had to override its search function to manually inject the keyword scoring system so it would sort the results properly.
2. **Agent Logic:** I gave ChatGPT my ASCII flow diagram and my planning loop breakdown. I expected it to build the `run_agent` function inside `agent.py` with the correct conditional structure. It did a great job setting up the session state and the conditional logic, but I had to intervene to fix an issue where it was passing the wrong variable names to the Gradio interface in `app.py`.
